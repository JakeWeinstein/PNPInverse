\section*{Weekly Focus}

This week's work builds on the completed Butler--Volmer (BV) forward solver
and I--V curve sweep (Feb~25 writeup) by adding an \textbf{adjoint-gradient
inference pipeline for recovering BV exchange rate constants~$k_0$ from
current-density curves}.  The new pipeline mirrors the existing Robin-kappa
inference architecture in \texttt{FluxCurve/} but is specialized for BV
physics, including warm-start voltage continuation and current-density
observables.  The implementation spans four phases: modifying the forward
solver for adjoint compatibility, adding BV steady-state helpers, creating
five new FluxCurve BV modules, and building a user-facing inference script.

The pipeline was then \textbf{tested end-to-end}, uncovering and fixing
three bugs (mixed-function warm-start, sign convention mismatch, and the
need for log-space optimization).  With these fixes, L-BFGS-B recovers
both $k_{0,1}$ and $k_{0,2}$ to $<3\%$ error from a $10\times$-wrong
initial guess in 6 iterations.  A \textbf{noise sensitivity study} at
0/2/5/10\% noise (10 total runs) confirmed that both parameters are
identifiable with errors scaling approximately linearly with noise level.

We then extended the pipeline to the \textbf{full 4-species charged system}
(O$_2$, H$_2$O$_2$, H$^+$, ClO$_4^-$), revealing $k_{0,2}$
identifiability limitations from total current density.  Next, we
implemented \textbf{transfer coefficient ($\alpha$) inference}---making
$\alpha$ adjoint-compatible via \texttt{fd.Function(R\_space)} and adding
\texttt{control\_mode} support throughout the pipeline.  Alpha inference
recovers both $\alpha_1$ and $\alpha_2$ to $<0.5\%$ error.  Finally,
\textbf{joint $(k_0, \alpha)$ inference} with a 4-element control vector
achieves $<2.5\%$ error on $\alpha$ and ${\sim}20\%$ on $k_0$ (reflecting
$k_0$--$\alpha$ correlation).

Most recently, we developed a \textbf{POD/RBF surrogate model} (Section~29)
that replaces PDE solves with RBF interpolation during inference, reducing
optimization time from ${\sim}345$\,s to $< 0.1$\,s ($> 3000\times$
speedup; target data generation of ${\sim}70$\,s is a shared fixed cost
excluded from this comparison).  Three rounds of iterative development
(v8, v8.1, v9) revealed that the surrogate acts as an implicit regularizer,
and that increasing training data yields diminishing returns.


%% ============================================================
\section*{1. Motivation}
%% ============================================================

The existing \texttt{FluxCurve/} pipeline can recover Robin boundary
transfer coefficients~$\kappa$ from steady-state flux curves using
firedrake-adjoint and L-BFGS-B.  The BV forward solver
(\texttt{Forward/bv\_solver.py}) generates physically realistic I--V curves
with coupled electrochemical reactions, but until now there was no way to
\emph{infer} the kinetic parameters from data.

The exchange rate constant~$k_0$ is the most natural first target for
inference:
\begin{itemize}
  \item $k_0$ scales the BV reaction rate linearly ($R_j = k_{0,j} \cdot [\ldots]$),
        making the objective landscape well-behaved---directly analogous to
        Robin~$\kappa$.
  \item $k_0$ is the primary fitting parameter in Tafel analysis and the most
        experimentally relevant BV parameter for characterizing catalyst activity.
  \item For the 2-species model (O$_2 \to$ H$_2$O$_2$, H$_2$O$_2 \to$ H$_2$O),
        we infer $[k_{0,1}, k_{0,2}]$---a 2-element control vector identical in
        structure to Robin~$[\kappa_0, \kappa_1]$.
\end{itemize}

The transfer coefficient~$\alpha$ (which controls I--V curve shape rather
than amplitude) has a more nonlinear landscape and is planned as a follow-up.


%% ============================================================
\section*{2. Phase 1: Adjoint-Compatible $k_0$ in the Forward Solver}
%% ============================================================

\subsection*{2.1 Problem}

Firedrake-adjoint's \texttt{adj.Control()} requires \texttt{fd.Function}
objects on the adjoint tape.  The BV forward solver previously stored $k_0$
values as \texttt{fd.Constant}, which are invisible to the tape and cannot
serve as optimization controls.

\subsection*{2.2 Solution}

In \texttt{Forward/bv\_solver.py}, function \texttt{build\_forms()}, we
changed $k_0$ from \texttt{fd.Constant} to \texttt{fd.Function(R\_space)}
(a single-DOF ``Real'' function space) in both code paths:

\begin{itemize}
  \item \textbf{Multi-reaction path} (lines 608--611): Each reaction's
        $k_{0,j}$ is now
        \texttt{fd.Function(R\_space, name=f"bv\_k0\_rxn\{j\}")}, assigned
        via \texttt{.assign(float(rxn["k0\_model"]))}.
  \item \textbf{Legacy per-species path} (lines 646--651): Same pattern
        with \texttt{name=f"bv\_k0\_sp\{i\}"}.
\end{itemize}

All $k_0$ functions are accumulated into a list \texttt{bv\_k0\_funcs} and
stored in the context dict alongside the existing \texttt{bv\_rate\_exprs}.
This is safe because \texttt{fd.Function(R\_space)} with one DOF behaves
identically to \texttt{fd.Constant} in UFL expressions.  The Robin solver
uses the same pattern with \texttt{kappa\_funcs}.

\subsection*{2.3 Verification}

A quick functional test confirmed:
\begin{itemize}
  \item \texttt{bv\_k0\_funcs} has 2 entries, both \texttt{fd.Function}
        with correct values ($\hat{k}_{0,1} \approx 0.00114$,
        $\hat{k}_{0,2} \approx 4.76 \times 10^{-5}$).
  \item The SNES solver converges in 3 time steps at $\hat\eta = -5$.
  \item Assembled reaction rates $R_0$, $R_1$ match pre-modification values.
\end{itemize}

\section*{3. Phase 2: BV Steady-State Helpers}
%% ============================================================

Five new functions were added to \texttt{Forward/steady\_state.py},
following the existing Robin helper patterns:

\begin{enumerate}
  \item \texttt{configure\_bv\_solver\_params(base, phi\_applied, k0\_values)}
        --- deep-copies the parameter list, sets \texttt{params[7]} and
        \texttt{params[9]} to the applied voltage, and injects $k_0$ values
        into the \texttt{bv\_bc} config (both multi-reaction
        \texttt{reactions[j].k0} and legacy \texttt{bv\_bc.k0} paths).

  \item \texttt{compute\_bv\_reaction\_rates(ctx)} --- assembles
        $\int R_j \, ds(\text{electrode})$ for each reaction expression in
        \texttt{ctx["bv\_rate\_exprs"]}.  Returns a list of dimensionless
        rates.

  \item \texttt{compute\_bv\_current\_density(ctx, i\_scale)} --- calls
        \texttt{compute\_bv\_reaction\_rates} and converts to current
        density: $I = -(\sum_j R_j) \cdot \texttt{i\_scale}$.

  \item \texttt{solve\_bv\_to\_steady\_state\_for\_phi\_applied(...)} ---
        single-point BV forward solve with current-density-based
        steady-state detection.  Returns a \texttt{SteadyStateResult} with
        \texttt{observed\_flux} in current-density units.

  \item \texttt{sweep\_phi\_applied\_steady\_bv\_flux(...)} --- builds the
        BV context once and sweeps $\hat\eta$ values sequentially with
        \textbf{warm-start continuation}: each point's converged solution
        is carried forward as the IC for the next point.  This is essential
        for BV convergence at large overpotentials (see \S3.5 of the
        Feb~25 writeup).
\end{enumerate}

\section*{4. Phase 3: FluxCurve BV Modules}
%% ============================================================

Rather than refactoring the Robin-specific FluxCurve modules (which could
break the working pipeline), we created \textbf{five parallel BV-specific
modules} that reuse the solver-agnostic infrastructure
(\texttt{recovery.py}, \texttt{results.py}, \texttt{plot.py}).

\subsection*{4.1 \texttt{FluxCurve/bv\_config.py}}

Defines \texttt{BVFluxCurveInferenceRequest}, mirroring
\texttt{RobinFluxCurveInferenceRequest} with BV-specific fields:
\begin{itemize}
  \item \texttt{true\_k0} / \texttt{initial\_guess} instead of
        \texttt{true\_value}
  \item \texttt{k0\_lower} / \texttt{k0\_upper} bounds
  \item \texttt{current\_density\_scale}: conversion factor from
        dimensionless BV rate to physical current density
  \item \texttt{mesh\_Nx}, \texttt{mesh\_Ny}, \texttt{mesh\_beta}: graded
        rectangle mesh parameters
  \item \texttt{observable\_mode}: \texttt{"current\_density"} (default),
        \texttt{"peroxide\_current"}, or \texttt{"reaction"}
\end{itemize}

\subsection*{4.2 \texttt{FluxCurve/bv\_observables.py}}

Builds scalar UFL observable forms from BV reaction rate expressions:
\begin{itemize}
  \item \texttt{\_build\_bv\_observable\_form(ctx, mode, reaction\_index,
        scale)} constructs:
    \begin{itemize}
      \item \texttt{"current\_density"}: $\texttt{scale} \cdot \sum_j R_j
            \cdot ds(\text{electrode})$
      \item \texttt{"peroxide\_current"}: $\texttt{scale} \cdot (R_0 - R_1)
            \cdot ds(\text{electrode})$
      \item \texttt{"reaction"}: $\texttt{scale} \cdot R_j \cdot
            ds(\text{electrode})$
    \end{itemize}
  \item The \texttt{scale} parameter is an \texttt{fd.Constant} wrapping
        \texttt{i\_scale} and sign, so the full current-density observable
        is on the adjoint tape.
  \item \texttt{\_build\_bv\_scalar\_target\_in\_control\_space} and
        \texttt{\_bv\_gradient\_controls\_to\_array} mirror the Robin
        versions but use \texttt{ctx["bv\_k0\_funcs"]} instead of
        \texttt{ctx["kappa\_funcs"]}.
\end{itemize}

\subsection*{4.3 \texttt{FluxCurve/bv\_point\_solve.py}}

The key architectural difference from the Robin pipeline:

\begin{quote}
\textbf{Robin:} Each $\phi_\text{applied}$ point is solved independently
(optionally in parallel via \texttt{ProcessPoolExecutor}), starting from a
cold IC.

\textbf{BV:} Points are solved \textbf{sequentially} with warm-start
continuation---sorted by $|\hat\eta|$, each point's converged solution
carried forward as the IC for the next.  This is essential because BV
solves at large overpotentials are fragile from cold starts.
\end{quote}

The function \texttt{solve\_bv\_curve\_points\_with\_warmstart} implements
this:
\begin{enumerate}
  \item Sort $\phi_\text{applied}$ points by $|\hat\eta|$
  \item For each point (in voltage order):
    \begin{enumerate}
      \item Clear adjoint tape, enable annotation
      \item Build fresh context + forms (for clean tape)
      \item Restore last converged $U$ data as IC (warm-start)
      \item Assign current $k_0$ values to \texttt{bv\_k0\_funcs}
      \item Time-step to steady state (annotated)
      \item Assemble observable, compute
            $J = \tfrac{1}{2}(I_\text{sim} - I_\text{target})^2$
      \item Compute adjoint gradient via
            \texttt{ReducedFunctional.derivative()}
      \item Store result; snapshot $U$ for next point's warm-start
    \end{enumerate}
  \item Return results in original (unsorted) order
\end{enumerate}

Per-point recovery uses the same staged phases as Robin
(\texttt{recovery.py}): increase \texttt{snes\_max\_it}, anisotropy
reduction, tolerance relaxation.

\subsection*{4.4 \texttt{FluxCurve/bv\_curve\_eval.py}}

\texttt{evaluate\_bv\_curve\_objective\_and\_gradient} calls the warm-start
point solver and aggregates:
\begin{itemize}
  \item Total objective $= \sum_i J_i$ (only converged points)
  \item Total gradient $= \sum_i \nabla_{k_0} J_i$ (only converged points)
  \item Failed points get penalty objective and zero gradient
  \item Anisotropy recovery fallback if many points fail
\end{itemize}

\subsection*{4.5 \texttt{FluxCurve/bv\_run.py}}

Top-level orchestration:
\begin{itemize}
  \item \texttt{ensure\_bv\_target\_curve}: generates synthetic BV target
        I--V curve via \texttt{sweep\_phi\_applied\_steady\_bv\_flux} with
        true~$k_0$, adds Gaussian noise
  \item \texttt{run\_scipy\_bv\_adjoint\_optimization}: SciPy L-BFGS-B
        loop with cached evaluations (kappa-keyed at 12-digit precision),
        live plotting, CSV history logging
  \item \texttt{run\_bv\_k0\_flux\_curve\_inference}: top-level entry point
        that builds the graded mesh, generates/loads target data, runs
        optimization, saves outputs (CSV, PNG, optional GIF)
\end{itemize}

Reuses solver-agnostic utilities: \texttt{clip\_kappa},
\texttt{\_LiveFitPlot}, \texttt{add\_percent\_noise},
\texttt{write\_phi\_applied\_flux\_csv}.

\subsection*{4.6 \texttt{FluxCurve/\_\_init\_\_.py}}

Added two new public exports:
\begin{lstlisting}
from FluxCurve import BVFluxCurveInferenceRequest
from FluxCurve import run_bv_k0_flux_curve_inference
\end{lstlisting}

\section*{5. Phase 4: Inference Script}
%% ============================================================

\texttt{scripts/inference/Infer\_BVk0\_from\_current\_density\_curve.py}
is the user-facing entry point, following the pattern of the Robin
inference script.

\subsection*{5.1 Physical setup}

\begin{itemize}
  \item 2-species neutral model: O$_2$ ($D = \SI{2.1e-9}{m^2/s}$) and
        H$_2$O$_2$ ($D = \SI{1.6e-9}{m^2/s}$), $z = [0, 0]$
  \item $c_\text{bulk} = \SI{0.5}{mol/m^3}$,
        $L_\text{ref} = \SI{100}{\micro m}$
  \item R$_1$: $k_{0,1} = \SI{2.4e-8}{m/s}$, $\alpha_1 = 0.627$
        (reversible O$_2 \to$ H$_2$O$_2$)
  \item R$_2$: $k_{0,2} = \SI{1e-9}{m/s}$, $\alpha_2 = 0.5$
        (irreversible H$_2$O$_2 \to$ H$_2$O)
\end{itemize}

\subsection*{5.2 Mesh and boundary markers}

Uses \texttt{make\_graded\_rectangle\_mesh(Nx=4, Ny=200, beta=3.0)}---a
power-law graded 2D mesh that clusters nodes near the electrode at $y=0$.

\begin{center}
\begin{tabular}{ll}
\toprule
Marker & Boundary \\
\midrule
3 & Bottom ($y=0$): electrode (BV BC + Dirichlet $\phi$) \\
4 & Top ($y=1$): bulk (Dirichlet $c_i$, ground $\phi = 0$) \\
1, 2 & Left/Right: zero-flux (natural BC) \\
\bottomrule
\end{tabular}
\end{center}

Note: these markers differ from \texttt{bv\_iv\_curve.py} which uses
\texttt{UnitSquareMesh(32,32)} with electrode at marker~1 and bulk at
marker~3.

\subsection*{5.3 All 7 convergence strategies}

\begin{enumerate}
  \item Voltage continuation with warm-start (sweep $\hat\eta$ from 0 to
        target, reuse previous solution)
  \item Inner time-stepping / pseudo-transient continuation (BDF-1 until
        L2 change $<$ tol)
  \item Exponent clipping at $\pm 50$
  \item Concentration floor $\varepsilon_c = 10^{-12}$
  \item Dirichlet-decoupled BV exponent (\texttt{use\_eta\_in\_bv = True})
  \item L2 linesearch with $\lambda_\text{max} = 0.5$
  \item MUMPS direct solver with auto-scaling
        (\texttt{icntl\_8 = 77}, \texttt{icntl\_14 = 80})
\end{enumerate}

\subsection*{5.4 Optimizer configuration}

\begin{itemize}
  \item 15-point sweep: $\hat\eta$ from $-1$ to $-20$ (in thermal voltage
        units)
  \item Observable: total current density
        $I = -(\sum_j R_j) \cdot I_\text{scale}$ in mA/cm$^2$
  \item $I_\text{scale} = n_e F D_\text{ref} c_\text{bulk} / L_\text{ref}
        \times 0.1 \approx \SI{0.1013}{mA/cm^2}$
  \item True $k_0$: $[\hat{k}_{0,1}, \hat{k}_{0,2}]$ (known ground truth)
  \item Initial guess: $[5.0, 5.0]$ (deliberately wrong)
  \item Method: L-BFGS-B with box bounds $[10^{-8}, 100]$
  \item Up to 30 iterations, \texttt{gtol} $= 10^{-6}$
  \item Output: \texttt{StudyResults/bv\_k0\_inference/}
\end{itemize}

\section*{7. Key Design Decisions}
%% ============================================================

\begin{enumerate}
  \item \textbf{Parallel modules, not refactored abstractions.}  The Robin
        FluxCurve modules have hardcoded imports to
        \texttt{robin\_solver.build\_context}, \texttt{kappa\_funcs}, etc.
        Rather than introducing a solver-agnostic abstraction layer (risky,
        could break the working Robin pipeline), we created parallel
        BV-specific modules (\texttt{bv\_config.py},
        \texttt{bv\_observables.py}, \texttt{bv\_point\_solve.py},
        \texttt{bv\_curve\_eval.py}, \texttt{bv\_run.py}) that reuse only
        the genuinely solver-agnostic utilities (\texttt{recovery.py},
        \texttt{results.py}, \texttt{plot.py}).

  \item \textbf{Sequential warm-start, not parallel points.}  The Robin
        pipeline can solve each $\phi_\text{applied}$ point independently
        because Robin boundary conditions are mild.  BV solves at large
        $|\hat\eta|$ require careful voltage continuation from equilibrium.
        Trading parallelism for convergence robustness is essential.

  \item \textbf{No replay mode.}  Replay (caching ReducedFunctionals across
        $k_0$ updates) was disabled in the Robin pipeline due to invalid
        non-steady evaluations.  We skip it entirely for BV.

  \item \textbf{Current density as the observable.}  The optimizer works
        directly in mA/cm$^2$ units for both target and simulated curves.
        The conversion factor (\texttt{i\_scale}) is passed as an
        \texttt{fd.Constant} in the UFL form so the adjoint tape sees the
        full chain from $k_0$ to current density.
\end{enumerate}


%% ============================================================
\section*{8. End-to-End Test}
%% ============================================================

Running \texttt{Infer\_BVk0\_from\_current\_density\_curve.py} in the
Firedrake environment uncovered three bugs that were fixed before the
pipeline converged successfully.

\subsection*{8.1 Bug 1: Mixed function data handling for warm-start}

\textbf{Symptom:} \texttt{AttributeError: 'tuple' object has no attribute
'copy'} at \texttt{bv\_point\_solve.py:230}.

\textbf{Cause:} For a \texttt{MixedFunctionSpace} (2 species + potential),
\texttt{U.dat.data\_ro} returns a tuple of arrays (one per subfunction),
not a single array.  The warm-start snapshot code assumed a single array.

\textbf{Fix:} Changed the snapshot/restore operations:
\begin{lstlisting}
# Snapshot (line 230):
carry_U_data = tuple(d.data_ro.copy() for d in U.dat)

# Restore (line 135):
for src, dst in zip(carry_U_data, ctx["U"].dat):
    dst.data[:] = src
\end{lstlisting}

\subsection*{8.2 Bug 2: Sign mismatch between target and observable}

\textbf{Symptom:} Optimizer converges to wrong answer.  Inspection
revealed that the target I--V curve had opposite sign to the simulated
observable at each voltage point.

\textbf{Cause:} Two independent sign conventions collided:
\begin{itemize}
  \item \texttt{compute\_bv\_current\_density} computes
        $I = -(\sum_j R_j) \cdot \texttt{i\_scale}$ (built-in negation
        for cathodic convention).
  \item \texttt{\_build\_bv\_observable\_form} computes
        $\texttt{scale} \cdot \sum_j R_j \cdot ds$ (no built-in negation).
\end{itemize}
When both receive \texttt{scale = $-I_\text{scale}$}, the target becomes
positive (double negation: $-(-) = +$) while the simulated observable is
negative ($(-I_\text{scale}) \cdot R > 0$ for cathodic $R < 0$, but the
net sign depends on which function applies the negation).

\textbf{Fix:} In \texttt{ensure\_bv\_target\_curve}, pass
\texttt{i\_scale = $-$current\_density\_scale} to the sweep function,
canceling the built-in negation in \texttt{compute\_bv\_current\_density}
so that both target and observable use the same sign convention.

\subsection*{8.3 Bug 3: Log-space optimization}

\textbf{Symptom:} With the sign bug fixed, the optimizer reports
``CONVERGENCE'' but produces $k_{0,1}$ with 91\% error and $k_{0,2}$
with $48{,}000\times$ error.  SciPy found a local minimum, not the
global one.

\textbf{Cause:} The true values ($\hat{k}_{0,1} \approx 0.00114$,
$\hat{k}_{0,2} \approx 4.76 \times 10^{-5}$) span two orders of
magnitude, and the initial guess $[5.0, 5.0]$ is ${\sim}4{,}400\times$
too high for $k_{0,1}$ and ${\sim}100{,}000\times$ too high for
$k_{0,2}$.  L-BFGS-B in linear space cannot efficiently traverse this
scale gap because:
\begin{itemize}
  \item A single L-BFGS-B step in $k_0$-space produces additive updates
        $\Delta k_0$, which cannot reduce a parameter by orders of
        magnitude without many iterations.
  \item The Hessian approximation becomes ill-conditioned when control
        variables differ by $10^4\times$.
\end{itemize}

\textbf{Fix:} Implemented \textbf{log-space optimization}.  The optimizer
works in $x = \log_{10}(k_0)$ space, with the gradient transformed via
the chain rule:
\begin{equation}
  \frac{\partial J}{\partial x_j}
  = \frac{\partial J}{\partial k_{0,j}} \cdot k_{0,j} \cdot \ln 10
\end{equation}
This makes multiplicative updates natural: a step $\Delta x = 1$
corresponds to a $10\times$ change in $k_0$.  Added to
\texttt{BVFluxCurveInferenceRequest} as \texttt{log\_space: bool = True}
(default).  The initial guess was also changed from $[5.0, 5.0]$ to
$[0.01, 0.001]$ (${\sim}10\times$ too high, within 1--2 orders of
magnitude).

\subsection*{8.4 Converged result}

With all three fixes applied, the optimizer converges in \textbf{6
L-BFGS-B iterations} (7 function evaluations), all 15 voltage points
converging at every evaluation.  Synthetic data generated with 2\% noise
($\sigma = 0.02 \times \mathrm{RMS}$, seed 20260226).

(clean and noisy), and Figure~\ref{fig:target-vs-fit} compares the
target data, initial guess, and optimized fit.

\section*{9. Noise Sensitivity Study}
%% ============================================================

\subsection*{9.1 Motivation}

The end-to-end test (Section~8) demonstrated that the pipeline can
recover both $k_{0,1}$ and $k_{0,2}$ from data with 2\% noise.  A
natural follow-up question is: \emph{how does recovery accuracy degrade
as noise increases?}  This is important for two reasons:
\begin{enumerate}
  \item \textbf{Identifiability:} If one parameter's error grows much
        faster than the other's, it indicates a structural sensitivity
        difference---one parameter may be poorly identifiable from
        current-density data alone.
  \item \textbf{Experimental design:} Understanding the noise--error
        relationship sets expectations for what measurement precision is
        needed to achieve a target accuracy on $k_0$.
\end{enumerate}

\subsection*{9.2 Methods}

The study script
\texttt{scripts/studies/bv\_k0\_noise\_sensitivity.py} tests 4 noise
levels: $\{0, 2, 5, 10\}\%$.  At each nonzero noise level, 3
independent random seeds are used (seeds 20260304, 20261304, 20262304);
the 0\% level uses a single run (noise-free data is deterministic).
This yields $1 + 3 \times 3 = 10$ total optimization runs.

Each run uses the same configuration as the end-to-end test:
\begin{itemize}
  \item 15-point sweep: $\hat\eta \in [-1, -20]$
  \item Log-space optimization with initial guess
        $[0.01, 0.001]$ (${\sim}10\times$ too high)
  \item L-BFGS-B with up to 50 iterations, \texttt{gtol}~$= 10^{-6}$
  \item Noise model: Gaussian with
        $\sigma = (\text{noise\%}/100) \times \mathrm{RMS}(\text{clean curve})$
  \item Live plotting disabled; forward recovery enabled
\end{itemize}

All 10 runs completed.  9 of 10 reported SciPy optimization success;
one run (5\% noise, seed 20260304) returned ABNORMAL status but still
produced reasonable estimates (18\% and 11\% error).

\section*{11. Transfer Coefficient ($\alpha$) Inference}
%% ============================================================

\subsection*{11.1 Motivation}

With $k_0$ inference validated, we extended the pipeline to infer the BV
transfer coefficient~$\alpha$, which controls the \textbf{shape} of the
I--V curve (as opposed to its amplitude).  The transfer coefficient appears
in the BV exponentials:
\begin{equation}
  R_j = k_{0,j}\bigl[c_\text{red} \exp(-\alpha_j \hat\eta / n_e)
        - c_\text{ox} \exp((1-\alpha_j)\hat\eta / n_e)\bigr]
\end{equation}
and therefore affects the gradient more nonlinearly than $k_0$.

\subsection*{11.2 Implementation}

Three changes were required to make $\alpha$ adjoint-compatible:

\begin{enumerate}
  \item \textbf{Forward solver} (\texttt{Forward/bv\_solver.py}): Changed
        $\alpha$ from \texttt{fd.Constant} to \texttt{fd.Function(R\_space)},
        mirroring the $k_0$ change.  Added \texttt{bv\_alpha\_funcs} to
        the context dict.

  \item \textbf{Steady-state helpers} (\texttt{Forward/steady\_state.py}):
        Extended \texttt{configure\_bv\_solver\_params} to accept an
        optional \texttt{alpha\_values} parameter, injecting into both
        \texttt{reactions[j].alpha} and legacy \texttt{bv\_bc.alpha}.

  \item \textbf{FluxCurve modules}: Added \texttt{control\_mode} parameter
        throughout the pipeline:
        \begin{itemize}
          \item \texttt{"k0"}: optimize $k_0$ only (existing behavior)
          \item \texttt{"alpha"}: optimize $\alpha$ only ($k_0$ fixed)
          \item \texttt{"joint"}: optimize both $k_0$ and $\alpha$
        \end{itemize}
        When \texttt{control\_mode="alpha"}, the adjoint controls are
        \texttt{bv\_alpha\_funcs} instead of \texttt{bv\_k0\_funcs}.
        Alpha uses \textbf{linear-space} optimization (not log-space)
        since $\alpha \in (0, 1)$.

  \item \textbf{Config} (\texttt{FluxCurve/bv\_config.py}): Added fields
        \texttt{control\_mode}, \texttt{true\_alpha},
        \texttt{initial\_alpha\_guess}, \texttt{alpha\_lower/upper},
        \texttt{alpha\_log\_space}, \texttt{fixed\_k0}.
\end{enumerate}

%% ============================================================
\section*{20A. Inference Pipeline Optimization}
%% ============================================================

\subsection*{20A.1 Bottleneck Analysis}

Profiling the charged joint inference revealed that each optimizer
evaluation took ${\sim}90$\,s (10 voltage points $\times$ ${\sim}9$\,s
each).  A 5-lambda regularization study with 30 iterations per lambda
would require ${\sim}2$--3 hours.  Detailed instrumentation identified
three dominant costs:

\begin{center}
\begin{tabular}{lrr}
\toprule
Component & Time/eval & Fraction \\
\midrule
Adjoint tape replay (\texttt{rf(control\_state)}) & ${\sim}18$\,s & 39\% \\
Forward PTC solves (8--11 steps/point) & ${\sim}21$\,s & 46\% \\
Adjoint linear solves (\texttt{rf.derivative()}) & ${\sim}6$\,s & 13\% \\
Setup (context/forms/solver build) & ${\sim}0.5$\,s & 1\% \\
\bottomrule
\end{tabular}
\end{center}

The key insight: \texttt{ReducedFunctional.\_\_call\_\_(control\_state)}
performs a \textbf{full tape replay}, re-solving every SNES step recorded
during the forward solve.  Since the tape already contains the correct
intermediate states from the forward solve that was just recorded, this
replay is entirely redundant---the objective value was already computed
during the forward pass.

\subsection*{20A.2 Optimizations Implemented}

Five optimizations were applied to
\texttt{FluxCurve/bv\_point\_solve.py} and supporting files:

\begin{enumerate}
  \item \textbf{P1: Eliminate tape replay.}  Replaced
        \texttt{float(rf(control\_state))} with
        \texttt{float(point\_objective)} (the already-computed value)
        and called \texttt{rf.derivative()} directly.  This skips the
        redundant re-forward-solve, cutting adjoint cost from
        ${\sim}2.5$\,s to ${\sim}0.35$\,s per point.

  \item \textbf{P2: Cross-evaluation warm-start.}  The first voltage
        point was previously cold-started every evaluation.  Now the
        converged solution from the previous evaluation is cached
        module-level and reused, since adjacent optimizer iterations
        have similar parameters.

  \item \textbf{P4: Linear predictor step.}  For warm-started points
        with two prior converged solutions, the initial condition is
        computed via linear extrapolation in voltage space:
        $U_\text{predict} = U_{k-1} + \frac{\eta_k - \eta_{k-1}}
        {\eta_{k-1} - \eta_{k-2}} (U_{k-1} - U_{k-2})$.
        This reduces PTC steps from 8--11 to 7--9 per point.

  \item \textbf{P5: Jacobian lagging.}  Added
        \texttt{snes\_lag\_jacobian: 2} and
        \texttt{snes\_lag\_jacobian\_persists: True} to reuse the
        MUMPS LU factorization every other Newton iteration.

  \item \textbf{P6: Adaptive pseudo-timestep (SER).}  Switched
        Evolution Relaxation grows \texttt{dt} when the observable
        change is decreasing and shrinks it when increasing.
        \texttt{dt\_model} was changed from a Python float to
        \texttt{fd.Constant} in \texttt{Forward/bv\_solver.py} to
        enable in-loop updates without form recompilation.
\end{enumerate}

Additional changes: default \texttt{gtol} relaxed from $10^{-4}$ to
$10^{-3}$ (noise floor is ${\sim}10^{-3}$ with 2\% data noise);
warm-start step cap set to 15 (from 200) for warm-started points.

\section*{20C. Steric Parameter Inference (Phase~4)}
%% ============================================================

The Bikerman steric exclusion model adds a chemical potential term
$\mu_{\text{steric}} = \ln\!\bigl(1 - \sum_j a_j c_j\bigr)$ to the

  \item Bridge points solved under

        each, ${\sim}0.2$\,s per bridge
  \item Carry \texttt{U} data and predictor state forward to next
        inference point
  \item 7 bridge points per evaluation (at $\hat\eta = -15.0, -19.5,
        -25.0, -30.3, -32.7, -38.0, -43.8$)
  \item \textbf{Zero failures} across 2046 total bridge evaluations

\section*{20E. Symmetric Voltage Range Inference (Phase~6)}
%% ============================================================

Phase~6 tests the physics prediction that including \textbf{anodic
(positive) overpotentials} alongside cathodic data will break the
$k_0$--$\alpha$ correlation by providing an independent constraint
on~$\alpha$ via the $(1-\alpha)$ Tafel slope.

\subsection*{20E.1 Physics Motivation}

In the Butler--Volmer rate expression, the cathodic branch depends
on~$\alpha$ through $\exp(-\alpha \hat\eta)$, so the cathodic Tafel
slope is $-\alpha$.  The anodic branch depends on
$\exp((1-\alpha)\hat\eta)$, giving the anodic Tafel slope
$+(1-\alpha)$.  With cathodic data alone, $k_0$ and~$\alpha$ are
correlated: many $(k_0, \alpha)$ pairs produce similar cathodic
curves.  Adding anodic data introduces an independent constraint
on~$\alpha$ through the $(1-\alpha)$ slope, potentially breaking the
correlation.

\subsection*{20E.2 Two-Branch Sweep Implementation}

A naive interleaved voltage sweep (ascending $|\hat\eta|$,
alternating cathodic and anodic) fails because warm-starting from a
cathodic solution to an anodic point produces non-physical initial
conditions---concentration profiles are inverted between the two
regimes.

\paragraph{Solution: two-branch sweep with hub state.}
Implemented in \texttt{bv\_point\_solve.py} via the function
\texttt{\_build\_sweep\_order()}:

\begin{enumerate}
  \item \textbf{Branch~1 (cathodic):} Solve all $\hat\eta < 0$
        points in order of ascending $|\hat\eta|$ (from
        $-0.25$ to $-28.0$), with standard warm-start continuation.
        After the point closest to equilibrium ($\hat\eta = -0.25$),
        save the converged solution as the \emph{hub state}.
  \item \textbf{Branch~2 (anodic):} Reset to the hub state and
        solve all $\hat\eta > 0$ points in order of ascending
        $\hat\eta$ ($+0.5, +1.0, +2.0, +3.0, +5.0$), warm-starting
        each from the previous anodic solution.
  \item Both branches maintain predictor state and bridge-point
        insertion independently.
\end{enumerate}

The hub state mechanism ensures that anodic solves start from a
near-equilibrium solution (low concentration gradients, small
electric field perturbation), which is physically compatible with
positive overpotentials.

\section*{21. Multi-Observable Joint Fitting}
%% ============================================================

\subsection*{21.1 Motivation}

All previous inference methods used a single observable per voltage
point---either total current density $I_{\text{tot}} = -(R_1 + R_2)
\cdot I_{\text{scale}}$ or peroxide current $I_{\text{pxd}} = -(R_1 -
R_2) \cdot I_{\text{scale}}$.  In both cases, the optimizer sees only
one scalar measurement per voltage, which provides one equation in four
unknowns ($k_{0,1}$, $k_{0,2}$, $\alpha_1$, $\alpha_2$).  The R2
parameters are in the null space because the R2 contribution to either
observable alone is small relative to R1 ($k_{0,2}/k_{0,1} \approx
0.04$).

The key insight is that the two observables provide \emph{linearly
independent} constraints on $R_1$ and $R_2$:
\begin{align}
  I_{\text{tot}} &= -(R_1 + R_2) \cdot I_{\text{scale}}, \\
  I_{\text{pxd}} &= -(R_1 - R_2) \cdot I_{\text{scale}}.
\end{align}
These two equations can be solved for $R_1$ and $R_2$ individually at
each voltage point, breaking the degeneracy.

\subsection*{21.2 Implementation}

The multi-observable evaluator modifies
\texttt{FluxCurve/bv\_curve\_eval.py} to compute and accumulate
\emph{two} objective terms per voltage point:
\begin{equation}
  J = \sum_{i} \bigl[
    w_1 \tfrac{1}{2}(I_{\text{tot},i} - I_{\text{tot},i}^*)^2
    + w_2 \tfrac{1}{2}(I_{\text{pxd},i} - I_{\text{pxd},i}^*)^2
  \bigr],
\end{equation}
where $w_1 = w_2 = 1$ (equal weighting).  The gradient is the sum of
the adjoint gradients from both terms.

The pipeline changes were:
\begin{itemize}
  \item \textbf{Target generation:} Two target curves are generated
        using \texttt{observable\_mode="current\_density"} and
        \texttt{observable\_mode="peroxide\_current"}, producing
        matched target arrays $I_{\text{tot}}^*$ and $I_{\text{pxd}}^*$.
  \item \textbf{Evaluator:} Each voltage point solves the forward
        problem once, then assembles both observable forms and computes
        two per-point objectives and their adjoint gradients.  The
        gradients are accumulated into the same control vector.
  \item \textbf{Adjoint tape:} Both observable forms share the same
        forward solve on the adjoint tape, so only one forward solve
        (and one tape replay) is needed per point---the cost per
        evaluation is approximately the same as single-observable
        inference.
\end{itemize}

\section*{22. Multi-pH Inference}
%% ============================================================

\subsection*{22.1 Motivation}

The BV rate expression for the cathodic branch contains the
concentration factor $(c_{\text{H}^+}/c_\text{ref})^{n_{\text{H}^+}}$,
where $n_{\text{H}^+} = 2$ for both reactions.  Varying the bulk
H$^+$ concentration (i.e., changing pH) modifies the electrode surface
concentration of H$^+$, which scales the reaction rates differently
for R1 and R2 depending on their respective $k_0$ and $\alpha$ values.
Fitting I--V curves at multiple pH conditions with shared $(k_0,
\alpha)$ but pH-dependent $c_{\text{H}^+}$ bulk values therefore
provides additional constraints on R2.

\subsection*{22.2 Implementation}

The multi-pH evaluator extends the inference pipeline to loop over
multiple pH conditions within a single optimizer evaluation:
\begin{equation}
  J = \sum_{\text{pH}} \sum_{i \in \text{converged}} \tfrac{1}{2}
      (I_i^{\text{pH}} - I_i^{*,\text{pH}})^2.
\end{equation}

Pipeline changes:
\begin{itemize}
  \item \textbf{Condition setup:} Three H$^+$ bulk concentrations
        $\hat{c}_{\text{H}^+} = [0.4, 0.2, 0.1]$ (corresponding to
        the nondimensional concentrations used in the solver).  For
        each condition, the H$^+$ bulk and---crucially---the counterion
        (ClO$_4^-$) bulk must be updated to maintain electroneutrality.
  \item \textbf{Target generation:} Separate I--V target curves are
        generated for each pH condition using the true parameters.
  \item \textbf{Per-condition evaluation:} The solver parameters
        (species bulk concentrations) are reconfigured for each pH
        before running the warm-start sweep.  Objectives and gradients
        are accumulated across conditions.
  \item \textbf{Shared controls:} The $k_0$ and $\alpha$
        \texttt{fd.Function(R\_space)} objects are shared across all
        conditions; the optimizer updates them once per evaluation.
\end{itemize}

\subsection*{22.3 Bug Fix: Counterion Electroneutrality}

During initial testing, the multi-pH evaluator produced incorrect
results because changing $c_{\text{H}^+,\text{bulk}}$ without updating
$c_{\text{ClO}_4^-,\text{bulk}}$ broke electroneutrality.  The Poisson
equation requires $\sum_j z_j c_{j,\text{bulk}} = 0$; with
$z_{\text{H}^+} = +1$ and $z_{\text{ClO}_4^-} = -1$, we need
$c_{\text{ClO}_4^-} = c_{\text{H}^+}$ at the bulk boundary.  The fix
updates both bulk concentrations in the solver parameter list when
switching pH conditions.

An additional issue arose with the original pH conditions (pH~0, 1, 2,
corresponding to $c_{\text{H}^+} = 1.0, 0.1, 0.01$\,M).  The highest
concentration ($c_{\text{H}^+} = 1.0$\,M) caused Debye-length stiffness
that prevented solver convergence.  The conditions were moderated to
$\hat{c}_{\text{H}^+} = [0.4, 0.2, 0.1]$, which are still
sufficiently spread to provide pH diversity.

        Combining multi-observable + multi-pH + symmetric voltage range

        require significant computational resources ($3 \times 15 \times
        2$ observable evaluations per optimizer step).
\end{enumerate}

\section*{23. Forward Solver Robustness Improvements}
%% ============================================================

\subsection*{23.1 Root Cause Analysis}

During multi-pH target generation with the original solver configuration
($8 \times 200$ mesh, $\beta = 3.0$, $\Delta t = 1.0$), 22\% of
voltage--condition combinations failed to converge (8/36 points across
3~pH conditions).  Investigation identified three distinct failure modes:

\begin{enumerate}
  \item \textbf{Debye-layer stiffness at elevated H$^+$.}  At
        $\hat{c}_{\mathrm{H}^+} = 0.4$, the Debye length shrinks,
        creating boundary layers that are unresolvable on the original
        mesh.  The PTC solver diverges with DIVERGED\_DTOL before reaching
        steady state.

  \item \textbf{BV rate explosion at large $|\hat\eta|$.}  The
        Butler--Volmer exponential $\exp(-\alpha \hat\eta)$ grows
        rapidly at large overpotentials.  Without regularization, this
        produces BV fluxes that exceed the solver's linear algebra
        tolerance, triggering DIVERGED\_LINEAR\_SOLVE (MUMPS failures).

  \item \textbf{Warm-start failure cascade.}  The voltage continuation
        scheme solves points in order of increasing $|\hat\eta|$, with
        each solve initialized from the previous converged solution.
        When a mid-sweep point fails, all subsequent points inherit a
        non-converged initial state, causing a cascade of failures that
        can wipe out an entire sweep branch.
\end{enumerate}

\subsection*{23.2 Fixes Implemented}

\begin{itemize}
  \item \textbf{Mesh refinement:} $N_y = 200 \to 300$, grading
        $\beta = 3.0 \to 4.0$.  The finer mesh with stronger grading
        resolves the Debye layer at higher H$^+$ concentrations.

  \item \textbf{Smaller pseudo-time steps:} $\Delta t = 1.0 \to 0.1$,
        $\text{max\_steps} = 200 \to 500$.  Smaller steps improve PTC
        convergence at the cost of more iterations per solve.

  \item \textbf{Softplus regularization of BV exponential.}  The raw
        exponential $\exp(-\alpha \hat\eta)$ is replaced with a softplus
        approximation that caps the growth rate, preventing numerical
        overflow at extreme overpotentials.

  \item \textbf{Denser $\hat\eta$ sweep:} 12~points $\to$ 19~points,
        with $\text{max\_eta\_gap} = 1.5$.  Smaller voltage gaps between
        consecutive points reduce the perturbation to the warm-start
        initial guess, preventing failure cascades.

  \item \textbf{SNES linesearch tuning:}
        \texttt{snes\_linesearch\_maxlambda}~$= 0.3$.  Reducing the
        maximum linesearch step prevents the nonlinear solver from taking
        overly aggressive steps that leave the convergence basin.  The
        L2 linesearch was used throughout; the backtracking (bt)
        linesearch was tested but proved counterproductive, causing
        slower convergence and more SNES failures.

  \item \textbf{pH condition adjustment:}
        $\hat{c}_{\mathrm{H}^+} = 0.4$ reduced to $0.3$.  Despite all
        mesh and solver improvements, $\hat{c}_{\mathrm{H}^+} = 0.4$
        remained intractable at moderate-to-large overpotentials.  The
        reduction to $0.3$ provides a sufficient pH range while
        maintaining full convergence.
\end{itemize}

\subsection*{23.3 Results}

After applying all fixes, target generation converged on \textbf{100\%}
of voltage--condition combinations (57/57), up from 78\% (28/36) with
the original configuration.  The convergence improvement is summarized
below:

\begin{center}
\begin{tabular}{lcc}
\toprule
{Configuration} & {Converged / Total} & {Rate} \\
\midrule
Original ($8 \times 200$, $\Delta t = 1.0$, 12~pts) & 28/36 & 78\% \\
Improved ($8 \times 300$, $\Delta t = 0.1$, 19~pts) & 57/57 & 100\% \\
\bottomrule
\end{tabular}
\end{center}

The improved configuration increases wall time per target generation
from ${\sim}2$~min to ${\sim}5$~min due to the finer mesh and more
PTC steps, but this is acceptable given the elimination of convergence
failures.

\subsection*{23.4 Key Lessons}

\begin{enumerate}
  \item \textbf{L2 linesearch outperforms backtracking for PNP--BV.}
        The bt linesearch's Armijo condition requires sufficient decrease,
        which is overly conservative for the stiff PNP--BV system.
        L2 with a reduced \texttt{maxlambda} provides adequate
        step-size control without excessive conservatism.

  \item \textbf{Dense voltage sweeps prevent cascade failures.}
        The most impactful single fix was reducing the maximum
        $\hat\eta$ gap from ${\sim}3$ to $1.5$---this eliminated
        warm-start failures entirely and is cheaper than mesh refinement.

  \item \textbf{There is a hard H$^+$ concentration ceiling.}
        $\hat{c}_{\mathrm{H}^+} = 0.4$ remains fundamentally
        intractable with the charged PNP formulation.  Reaching higher
        concentrations requires either a neutral-species formulation
        or adaptive mesh refinement in the electric double layer.
\end{enumerate}

\section*{24. Master Inference Protocol}
%% ============================================================

\subsection*{24.1 Motivation}

Previous inference results showed that different methods excel at
different parameters: staged inference recovers R1 well but fails on R2;
symmetric data recovers $\alpha_1$ but not $k_0$; multi-observable
fitting recovers R2 but sacrifices R1 accuracy.  The master inference
protocol aims to combine these strengths into a single automated
pipeline that recovers all four BV parameters ($k_{0,1}$, $k_{0,2}$,
$\alpha_1$, $\alpha_2$) simultaneously.  Three major iterations
(v1--v3) were developed, each informed by failure-mode analysis of the
previous version.

\subsection*{24.2 v1 Pipeline (5-Phase) --- Failed on R2}

The initial master protocol used a 5-phase staged approach:

\begin{enumerate}
  \item \textbf{Phase 1: $\alpha$-only symmetric.}  Recover $\alpha_1$
        and $\alpha_2$ using symmetric (anodic + cathodic) voltage data
        with $k_0$ fixed at initial guesses.
        \textbf{Result:} $\alpha_1$ error = 0.03\% (excellent).

  \item \textbf{Phase 2: $k_0$-only cathodic.}  Fix $\alpha$ at Phase~1
        values and recover $k_{0,1}$ and $k_{0,2}$ from cathodic-only
        total current data.
        \textbf{Result:} $k_{0,1}$ error = 4.7\% (excellent), but
        $k_{0,2}$ driven to 98.8\% error.

  \item \textbf{Phases 3--5: Joint refinements.}  Various joint
        optimizations starting from Phase~2 values.
        \textbf{Result:} All failed because $k_{0,2}$ was poisoned
        by Phase~2.
\end{enumerate}

\textbf{Root cause of failure:} The $k_0$-only optimization in Phase~2
minimizes the total current residual, which is dominated by R1.  The
optimizer achieves this by driving $k_{0,2} \to 0$, effectively
eliminating R2's small contribution to the total current.  Once
$k_{0,2}$ is near zero, subsequent phases cannot recover because the
objective gradient with respect to $k_{0,2}$ is vanishingly small at
$k_{0,2} \approx 0$.  Additionally, forward solver failures occurred
at deep cathodic points when $k_{0,2}$ was far from truth, creating
NaN/penalty cascades that further corrupted the optimization.

\subsection*{24.3 v2 Pipeline (4-Phase) --- Partial Success, Phase 3 Failure}

The revised protocol removed the $k_0$-only phase entirely:

\begin{enumerate}
  \item \textbf{Phase 1: $\alpha$-only symmetric.}  Same as v1.
        Recover $\alpha$ from anodic + cathodic Tafel slopes.

  \item \textbf{Phase 2: Multi-observable joint from cold $k_0$.}
        Instead of a $k_0$-only phase, jump directly to multi-observable
        (total + peroxide current) joint $(k_0, \alpha)$ optimization.
        Crucially, $k_0$ starts from the cold initial guess (not from a
        $k_0$-only fit), so $k_{0,2}$ has not been driven to zero.
        \textbf{Result:} $k_{0,1}$: 18.1\%, $k_{0,2}$: 9.2\%,
        $\alpha_1$: 9.2\%, $\alpha_2$: 8.5\%.

  \item \textbf{Phase 3: Total-current regularization annealing.}
        Anneal Tikhonov regularization on total current only.
        \textbf{Result:} \textbf{Destroyed R2:}
        $k_{0,2}$ diverged to 137\%, $\alpha_2$ to 90\%.

  \item \textbf{Phase 4: Multi-pH.}  Multi-condition fitting.
        \textbf{Result:} Could not recover from Phase~3 corruption.
\end{enumerate}

\textbf{Root cause of Phase~3 failure:} The total-current-only
objective in Phase~3 triggered the same mechanism as v1's Phase~2: the
optimizer reduced the total current residual by adjusting R2 parameters
away from truth.  The key insight is that the ``$k_{0,2} \to 0$''
failure mode is not specific to $k_0$-only optimization---it occurs
in \emph{any} phase that uses total current as the sole observable
while jointly optimizing all four parameters.  This led to the
\textbf{``never total-current-only'' rule}: every phase that
jointly optimizes R1 and R2 parameters must include the peroxide
observable.

\subsection*{24.4 v3 R1-Only Refinement Attempt --- Failed}

An intermediate v3 attempt used Phase~3 to refine R1 parameters only
(with R2 frozen at Phase~2 values) on total-current data:

\begin{itemize}
  \item R2 parameters stayed frozen as designed.
  \item $k_{0,1}$ \textbf{worsened}: 18\% $\to$ 66\%.
  \item $\alpha_1$ \textbf{worsened}: 9\% $\to$ 25\%.
\end{itemize}

\textbf{Root cause:} The residual R2 error from Phase~2 biases the
total current at every voltage point.  When the optimizer fits R1
parameters to total-current data that contains this R2 bias, it
compensates by distorting R1 parameters.  This demonstrates that
total-current-only refinement is unreliable even when only R1
parameters are free, because the total current conflates R1 and R2
contributions.

\subsection*{24.5 v3 Final Pipeline (3-Phase, Multi-Obs Throughout)}

The final protocol enforces multi-observable fitting in every joint
optimization phase, eliminating the possibility of total-current-only
corruption:

\begin{enumerate}
  \item \textbf{Phase 1: $\alpha$-only symmetric.}  12-point symmetric
        range ($\hat\eta \in [-28, +5]$), $\alpha$-only optimization
        with $k_0$ fixed at initial guesses.
        \textbf{Result:} $\alpha_1$ error = 0.36\%.

  \item \textbf{Phase 2: Multi-obs shallow cathodic.}  10-point
        shallow cathodic range ($\hat\eta \in [-1, -13]$),
        multi-observable (total + peroxide) joint $(k_0, \alpha)$
        optimization.  $k_0$ starts from the cold initial guess.
        \textbf{Result:} All four parameters at 8--18\% error.

  \item \textbf{Phase 3: Multi-obs full cathodic.}  15-point full
        cathodic range ($\hat\eta \in [-1, -46.5]$), multi-observable
        joint optimization, warm-started from Phase~2 values.
        \textbf{Result:} Improved 3 of 4 parameters.
\end{enumerate}

The \textbf{shallow-then-full} strategy is critical: Phase~2 uses only
the shallow cathodic range ($|\hat\eta| \le 13$) to avoid solver
failures during early optimization when parameters are far from truth.
Phase~3 extends to the full cathodic range ($|\hat\eta| \le 46.5$),
adding the deep diffusion-limited plateau constraint once parameters
are in the correct basin of attraction.

        staging, warm-starting, or phase ordering can recover R2 from
        total current alone.  The peroxide current observable
        ($I_{\text{pxd}} = -(R_1 - R_2) \cdot \text{scale}$) is the
        only measurement that provides a direct constraint on R2's
        sign and magnitude.

  \item \textbf{Pipeline ordering is critical.}  The $k_0$-only phase

        poisoning $k_{0,2}$ beyond recovery.  The lesson is that any

        benefit to $\alpha_1$ but does not substantially change the

\end{enumerate}

The v3 master pipeline (\S24) runs in ${\sim}2075$\,s (${\sim}34.6$~min).
While acceptable for single runs, iterating on pipeline design and
running parameter sweeps benefits from shorter wall-clock times.  The
goal was to reduce runtime without degrading parameter recovery accuracy
(tolerance: $< 0.1\%$ relative change in any parameter error).

\subsection*{25.2 Optimizations Retained (v4)}

\textbf{1. Shared forward solutions for multi-observable phases
(${\sim}15\%$ speedup on multi-obs phases).}  The forward PDE
solution is identical for both observables (total current and peroxide
current)---only the post-solve extraction differs.  In v3, the
Firedrake tape was cleared between observables, forcing a full
re-solve.  In v4, cache clearing was removed so the checkpoint-restart
fast path works for the secondary observable.  This avoids redundant
SNES solves in Phases~2 and~3.

File changed: \texttt{FluxCurve/bv\_curve\_eval.py}.

\textbf{2. Predictor validation (${\sim}5\%$ speedup).}  The warm-start
predictor extrapolates DOFs from the previous two converged solutions.
When parameters are far from truth (early optimizer iterations), the
predictor can produce states that deviate $>10\times$ from the carry
state, causing SNES to waste 10--15 iterations before diverging and
falling back to the simple warm-start.  In v4, a pre-solve check
compares the predictor DOFs against the carry state; if the maximum
deviation exceeds $10\times$, the predictor is immediately reverted.
This reduced predictor failure retries from ${\sim}35\%$ to near-zero.

File changed: \texttt{FluxCurve/bv\_point\_solve.py}.

\subsection*{25.3 Optimizations Tried and Reverted}

Three more aggressive strategies were tested and reverted because they
degraded parameter accuracy by $> 2\%$:

\begin{enumerate}
  \item \textbf{Fewer voltage points} (10$\to$7 in Phase~2,
        15$\to$10 in Phase~3).  $k_{0,1}$ error jumped from 16.7\% to
        52\%.  Insufficient I--V curve sampling destroyed
        identifiability---the optimizer could not distinguish between
        parameter combinations with fewer data constraints.

  \item \textbf{Skip Phase~1} ($\alpha$-only symmetric).  $k_{0,1}$
        error jumped to 58\%.  Phase~1's $\alpha$ warm-start, though
        modest in the final result, provides a critical starting basin
        for Phase~2.  Without it, the optimizer converges to a
        qualitatively different local minimum.

  \item \textbf{Aggressive convergence tolerances}
        (\texttt{maxiter}~25, \texttt{gtol}~$10^{-5}$).  $k_{0,1}$
        error jumped to 55\%.  L-BFGS-B was terminating before
        convergence; the default \texttt{maxiter}~40 and
        \texttt{gtol}~$10^{-8}$ are near-minimal for this 4-parameter
        problem.
\end{enumerate}

\subsection*{25.5 Diminishing Returns Analysis}

The runtime breakdown reveals why further optimization is difficult
without architectural changes:

\begin{itemize}
  \item \textbf{85\% of runtime is PETSc SNES solves + Firedrake
        adjoint tape replays.}  These are already optimized by the
        checkpoint-restart fast path and predictor validation.  Further
        reduction requires either fewer solves (which breaks
        identifiability) or faster individual solves (which requires
        parallel hardware).

  \item \textbf{The optimizer needs 20--30 evaluations for 4
        parameters.}  This is near the minimum for L-BFGS-B on a
        4-parameter problem with the observed condition number.
        Gauss--Newton could reduce this, but requires second-order
        adjoint infrastructure.

  \item \textbf{Voltage point count is near-minimal.}  Fewer than
        10 points in the shallow range or 15 in the full range breaks
        identifiability (as demonstrated by the reverted experiments).

  \item \textbf{Further $>15\%$ speedup requires parallel MPI
        point-solves or GPU acceleration.}  Independent voltage points
        within a single objective evaluation could be solved in
        parallel across MPI ranks.  This would require restructuring
        the serial sweep loop in \texttt{FluxCurve/bv\_curve\_eval.py}
        into a distributed map-reduce pattern---a fundamental
        architectural change.
\end{itemize}

\section*{27. Parallelism Optimization of Master Pipeline (v5--v7)}
%% ============================================================

Following the 11.7\% speedup in Section~25, three rounds of deeper
optimization were applied to the 3-phase master inference pipeline,
achieving a cumulative \textbf{4.4$\times$ wall-clock speedup}
(1832\,s $\to$ 415\,s) while simultaneously \emph{improving} accuracy
on all four BV parameters.

\subsection*{27.1 Key Insight: Parallel Fast Path}

The BV flux-curve inference pipeline evaluates the objective function
by sweeping over voltage points in sequence.  The \emph{first}
evaluation must be sequential because warm-start continuation uses each
converged solution as the initial condition for the next voltage point.
However, once this first sweep completes, the converged solutions are
cached in a checkpoint dictionary keyed by~$\hat\eta$.  All subsequent
L-BFGS-B evaluations (with updated parameters) can therefore solve
each voltage point \textbf{independently} from its cached IC, with no
inter-point data dependency.

This observation enables process-level parallelism: after the first
sequential sweep, the ``fast path'' dispatches voltage-point
forward$+$adjoint solves to a pool of worker processes.

\subsection*{27.2 Round 1: Parallel Fast Path (v5)}

\textbf{Implementation.}  A \texttt{ProcessPoolExecutor} with
\texttt{spawn} context (following the existing Robin pipeline pattern
in \texttt{FluxCurve/point\_solve.py}) distributes independent
voltage-point solves across worker processes.

\textbf{New file:} \texttt{FluxCurve/bv\_parallel.py}, containing:
\begin{itemize}
  \item \texttt{BVParallelPointConfig} --- a frozen, pickle-serializable
        dataclass holding all mesh/physics/solver parameters needed to
        reconstruct a BV solve context in a worker process.
  \item \texttt{\_bv\_worker\_init(config\_bytes)} --- called once per
        worker at pool creation.  Sets \texttt{OMP\_NUM\_THREADS=1}
        (critical for avoiding thread contention within PETSc), imports
        firedrake, builds the mesh and function spaces.
  \item \texttt{\_bv\_worker\_solve\_point(task)} --- receives a
        7-element tuple (parameter values, voltage, target flux, cached
        IC data), builds the BV context and forms, loads the cached IC
        into a firedrake Function, performs the forward solve$+$adjoint,
        and returns the point objective value and gradient.
  \item \texttt{BVPointSolvePool} --- manages the
        \texttt{ProcessPoolExecutor} lifecycle (creation, submit, shutdown).
\end{itemize}

\textbf{Modified files:}
\begin{itemize}
  \item \texttt{FluxCurve/bv\_point\_solve.py} --- added
        \texttt{\_solve\_cached\_fast\_path\_parallel()}, module-level
        \texttt{\_parallel\_pool} variable, and
        \texttt{set\_parallel\_pool()}/\texttt{close\_parallel\_pool()}
        functions.  The fast-path dispatcher checks whether a parallel
        pool is active and the current observable mode matches the
        pool's config; if so, it submits all voltage points as futures
        and gathers results.
  \item \texttt{FluxCurve/bv\_config.py} --- added
        \texttt{parallel\_fast\_path: bool} and
        \texttt{parallel\_workers: int} fields to
        \texttt{BVFluxCurveInferenceRequest}.
  \item \texttt{FluxCurve/\_\_init\_\_.py} --- added parallel exports.
\end{itemize}

\subsection*{27.3 Round 2: Multi-Observable Workers (v6)}

\textbf{Key insight:}  Workers can compute \emph{both} adjoint
gradients (primary and secondary observable) after a single forward
PDE solve, completely eliminating the sequential secondary-observable
sweep.

\textbf{Implementation --- ``Two-tape-pass'' approach:}
\begin{enumerate}
  \item The forward PDE solves to convergence (unchanged).
  \item The adjoint tape is cleared.
  \item The BV context and forms are rebuilt from scratch.
  \item The converged IC is loaded into fresh firedrake Functions.
  \item A single SNES step is executed (converged state $\to$
        converged state, typically ${\sim}1$ Newton iteration with
        residual below tolerance).  This re-records the solve on a
        clean tape.
  \item The primary observable form is built and the adjoint is
        computed.
  \item Steps 2--6 are repeated for the secondary observable (with
        the appropriate observable-mode flag).
\end{enumerate}

This ``two-tape-pass'' pattern was encapsulated in a new function
\texttt{\_bv\_worker\_adjoint\_tape\_pass()} in
\texttt{bv\_parallel.py}.  The task tuple was extended from 7 to 9
elements to carry the secondary target flux and a multi-obs flag.

\textbf{Modified files:}
\begin{itemize}
  \item \texttt{FluxCurve/bv\_parallel.py} --- added

        \texttt{\_bv\_worker\_solve\_point()} to handle 9-element tasks.
  \item \texttt{FluxCurve/bv\_point\_solve.py} --- added
        \texttt{\_solve\_cached\_fast\_path\_parallel\_multi\_obs()}.
  \item \texttt{FluxCurve/bv\_curve\_eval.py} --- modified
        \texttt{evaluate\_bv\_multi\_observable\_objective\_and\_gradient()}
        to detect a multi-obs-capable pool and skip the second (sequential)
        evaluation call entirely.
\end{itemize}

sequential}: BV physics at large overpotentials diverges without
warm-start continuation from the previous voltage point.  The parallel
fast path is already at 9~workers on a 10-core machine.  Pool
initialization overhead is dominated by the firedrake import
(${\sim}3$\,s per worker) and mesh construction, which are
irreducible with the \texttt{spawn} context required by firedrake's
PETSc backend.

\textbf{Strategies assessed and rejected:}
\begin{itemize}
  \item \textbf{Worker solver reuse} (${\sim}5$--10\% potential gain):
        Reusing the firedrake solver object across evaluations within a
        worker would avoid form recompilation.  However, the adjoint
        tape requires fresh forms for each pass (the tape records
        operator assembly), making reuse incompatible with correct
        gradient computation.  High implementation complexity for low
        reward.

  \item \textbf{Multi-fidelity mesh} (${\sim}15$--20\% potential gain):
        Using a coarser mesh for early optimizer evaluations and
        switching to the fine mesh for final evaluations.  Cache
        invalidation on mesh switch negates the fast-path advantage
        (all cached ICs must be recomputed on the new mesh), and the
        fine-mesh sweep becomes the new bottleneck.

  \item \textbf{Speculative parallel first evaluation} (${\sim}10$--15\%
        potential gain):  Running the first sweep in parallel by
        launching all voltage points simultaneously from the
        equilibrium IC (instead of sequentially from warm-start).
        BV physics at $|\hat\eta| > 10$ diverges from the equilibrium
        IC without intermediate warm-start states; only 3--4 of 10
        points would converge, requiring a sequential fallback anyway.

  \item \textbf{MPI Ensemble parallelism} (${\sim}2$--$3\times$
        potential gain):  Distributing voltage-point solves across MPI
        ranks using firedrake's Ensemble parallelism.  Would require
        restructuring from a single-process
        \texttt{ProcessPoolExecutor} architecture to an
        \texttt{mpiexec}-launched distributed pattern.  Deferred to
        future work as a separate development effort.
\end{itemize}

\subsection*{27.9 Architecture Summary}

The parallelism architecture follows a three-tier pattern:

\begin{enumerate}
  \item \textbf{Outer loop} (L-BFGS-B optimizer): sequential parameter
        updates.  Unchanged.

  \item \textbf{Middle loop} (voltage-point sweep): sequential for the
        first evaluation (warm-start IC population), then parallel for
        all subsequent evaluations (fast-path from cached ICs).

  \item \textbf{Inner loop} (per-point solve): each worker performs a
        forward PDE solve followed by one or two adjoint tape passes
        (primary and optionally secondary observable).  Workers are
        fully independent after receiving their cached IC, parameters,
        and target flux.
\end{enumerate}

The \texttt{spawn} context is required (not \texttt{fork}) because
firedrake and PETSc maintain internal state that is not fork-safe.
Each worker sets \texttt{OMP\_NUM\_THREADS=1} to prevent PETSc's
internal threading from contending with the process-level parallelism.
Worker-to-main communication uses pickle serialization of numpy arrays
(firedrake Functions are converted to \texttt{.dat} arrays before
sending and reconstructed on receipt).


%% ============================================================
\section*{29. POD/RBF Surrogate Model (v8--v9)}
%% ============================================================

Following the 4.4$\times$ parallelism speedup (Section~27), a
fundamentally different acceleration strategy was explored: replacing
PDE forward solves with a radial basis function (RBF) surrogate that
maps BV kinetic parameters directly to I--V curve observables.  Three
rounds of development (v8, v8.1, v9) were completed, reducing the
optimization time from ${\sim}345$\,s to $< 0.1$\,s ($> 3000\times$
speedup) with complementary accuracy trade-offs relative to the
PDE-based v7 pipeline.

\subsection*{29.1 Surrogate Architecture}

The surrogate is a \emph{parameter-to-observable} interpolant, not a
full POD decomposition of the solution field.  For each voltage
point~$\hat\eta_i$ and observable type (current density or peroxide
current), the surrogate directly predicts the scalar observable value
given the four BV parameters $\theta = (\log_{10} k_{0,1},\,
\log_{10} k_{0,2},\, \alpha_1,\, \alpha_2)$.

\textbf{RBF interpolation.}  The surrogate uses
\texttt{scipy.interpolate.RBFInterpolator} with a thin-plate spline
kernel ($\phi(r) = r^2 \log r$) and linear polynomial tail
(\texttt{degree=1}).  The $k_0$ parameters are sampled and
interpolated in $\log_{10}$ space (matching the optimizer's
parameterization), and all four input dimensions are $z$-score
normalized (zero mean, unit variance) before interpolation.  Two
separate RBF models are built: one for current-density observables
and one for peroxide-current observables.

\textbf{Gradient computation.}  Gradients are computed via central
finite differences with step size $h = 10^{-4}$:
\[
  \frac{\partial f}{\partial \theta_j} \approx
  \frac{f(\theta + h\,e_j) - f(\theta - h\,e_j)}{2h}.
\]
Each gradient evaluation requires $2 \times 4 = 8$ surrogate
evaluations---each taking microseconds---making the total gradient
cost negligible ($< 1$\,ms).

\textbf{NaN handling.}  Training samples that fail to converge at
specific voltage points produce NaN values.  The surrogate masks
these entries: only converged (non-NaN) values contribute to
interpolation at each voltage point.  This means different voltage
points may have slightly different training set sizes.

\subsection*{29.2 Training Data Generation}

Training data are generated by calling the existing \texttt{FluxCurve}
pipeline to compute I--V curves at LHS-sampled parameter combinations.

\textbf{Sampling strategy.}  Parameters are sampled via Latin
hypercube sampling (LHS) over the following ranges:
\begin{center}
\begin{tabular}{llll}
\toprule
Parameter & Space & Lower & Upper \\
\midrule
$k_{0,1}$ & $\log_{10}$ & $-4.5$ & $-1.5$ \\
$k_{0,2}$ & $\log_{10}$ & $-6.0$ & $-2.5$ \\
$\alpha_1$ & linear & $0.3$ & $0.9$ \\
$\alpha_2$ & linear & $0.2$ & $0.8$ \\
\bottomrule
\end{tabular}
\end{center}
The true parameters fall well within these ranges.

\textbf{Checkpointing.}  Because each PDE solve takes
${\sim}60$--$70$\,s, the training data generator checkpoints results
every 5 samples to an NPZ file.  This enables resume-on-crash and
incremental training set expansion.  Each sample's I--V curve
(22~voltage points $\times$ 2~observables) plus the 4-element
parameter vector are stored.

\textbf{Convergence rates.}  Not all parameter combinations converge
at all voltage points.  Round~1 achieved 90.5\% overall convergence
(181/200 samples fully valid); Round~2B achieved 89\% (445/500).
Failed samples are retained in the training set with NaN masking for
the unconverged points.

\subsection*{29.3 Round 1: Initial Surrogate (v8)}

The initial implementation created six new modules in a
\texttt{Surrogate/} package (\texttt{sampling.py},
\texttt{surrogate\_model.py}, \texttt{training.py},
\texttt{objectives.py}, \texttt{validation.py}, \texttt{io.py})
and four scripts in \texttt{scripts/surrogate/}.

\textbf{Training.}  $N = 200$ LHS samples, 181 valid

\begin{itemize}

  \item Peroxide current: NRMSE = 0.54\%
\end{itemize}
The peroxide-current surrogate is ${\sim}9\times$ less accurate than
the current-density surrogate, reflecting the peroxide current's
stronger nonlinear dependence on $k_{0,2}$ and $\alpha_2$.

\subsection*{29.4 Round 2A: Quick Fixes (v8.1)}

Two targeted modifications were tested:

\textbf{Removal of Phase~3.}  Since the full cathodic surrogate
degraded results, v8.1 removed Phase~3 entirely and used Phase~2
as the final result.

\textbf{Optimizer bound clamping.}  The L-BFGS-B bounds were
restricted to the training data range to prevent extrapolation
outside the surrogate's domain of validity.

\textbf{PDE refinement phase.}  A new Phase~3 was added that takes
the surrogate-recovered parameters as a starting point and runs
PDE-based L-BFGS-B refinement.  \emph{This did not improve results}:
the PDE refinement converged to a different local minimum because the
PDE solver faithfully reproduces the noise realization, while the
surrogate implicitly smooths it.  The surrogate's smoothing acts as a
form of \textbf{implicit regularization}, analogous to early stopping
or spectral truncation in classical inverse problems.

\textbf{Results (v8.1).}  Total time: 177.8\,s, comprising
${\sim}75$\,s target generation, $< 0.1$\,s surrogate optimization,
and ${\sim}101$\,s PDE refinement (the added PDE phase dominates the
increase over v8).  Best-of-phase selection picked the surrogate
Phase~2, producing identical accuracy to v8 Phase~2---confirming
that PDE refinement does not help.

\subsection*{29.5 Round 2B: More Training Data (v9)}

\textbf{Expanded training set.}  $N = 500$ LHS samples (445 valid,
89\% convergence), including 100 focused samples near the true
parameters (tighter LHS ranges: $\pm 0.5$ in $\log_{10}$ for $k_0$,
$\pm 0.15$ for $\alpha$).  One-time offline training cost:
${\sim}9.5$~hours (amortized over all subsequent inference runs).

\textbf{Cross-validated smoothing.}  The RBF smoothing
parameter~$\epsilon$ was cross-validated over $\epsilon \in \{0,
10^{-6}, 10^{-4}, 10^{-2}, 0.1, 1.0\}$ using 5-fold CV.  Result:
$\epsilon = 0$ (exact interpolation) was optimal for both
current-density and peroxide-current surrogates, confirming that
the training data are smooth enough that no additional smoothing
is needed.

\textbf{Validation metrics.}  On the held-out validation set:
\begin{itemize}
  \item Current density: NRMSE = 0.42\% (worse than v8's 0.06\%
        due to larger parameter range coverage)
  \item Peroxide current: NRMSE = 11.4\% (significantly worse)
\end{itemize}
The peroxide-current NRMSE degradation is the bottleneck for
$k_{0,2}$ accuracy: the surrogate cannot reproduce the fine
structure of the peroxide current's dependence on R2 parameters.

\subsection*{29.6 Analysis and Limitations}

\textbf{Accuracy trade-offs.}  The surrogate improves $k_{0,1}$,
$\alpha_1$, and $\alpha_2$ relative to PDE-based v7, but degrades
$k_{0,2}$.  This is because:
\begin{itemize}
  \item The current-density surrogate is highly accurate (NRMSE
        $\le 0.42\%$), so parameters identified primarily from
        current density ($k_{0,1}$, $\alpha_1$) benefit from the
        surrogate's implicit regularization.
  \item The peroxide-current surrogate has much higher error (NRMSE
        up to 11.4\%), limiting the accuracy of $k_{0,2}$ and
        $\alpha_2$, which require the peroxide observable for
        identifiability.
\end{itemize}

\textbf{Implicit regularization.}  The surrogate smooths the
parameter-to-observable map by interpolating a finite training set.
This acts as a low-pass filter on the objective landscape, reducing
sensitivity to noise in the target data.  PDE refinement re-introduces
this noise sensitivity, which is why it does not improve (and can
worsen) the surrogate result.  This finding suggests that the
PDE-based pipeline may be slightly \emph{over-fitting} the noise,
and that the surrogate's accuracy on $k_{0,1}$, $\alpha_1$, and
$\alpha_2$ may be closer to the achievable optimum.

\textbf{Diminishing returns.}  The progression from 181 to 445
training samples yielded modest improvements at rapidly increasing
offline cost:
\begin{center}
\begin{tabular}{l S[round-precision=1] S[round-precision=1]
                  S[round-precision=1]}
\toprule
Metric & {v8 (181 samples)} & {v9 (445 samples)} & {$\Delta$} \\
\midrule
$k_{0,2}$ err (\%)      &  9.6 &  7.6 & -2.0 \\

Offline cost (hr, one-time) &  3.1 &  9.5 & {$\times 3.1$} \\
\bottomrule
\end{tabular}
\end{center}
The peroxide-current NRMSE actually \emph{worsened} in v9 (despite
more data) because the expanded parameter range increases the
interpolation challenge.  The $k_{0,2}$ improvement came from the
focused samples near the true parameters rather than from the
overall NRMSE improvement.

\subsection*{29.9 Conclusions}

The POD/RBF surrogate provides a viable alternative to PDE-based
inference when speed is the priority.  Key findings:
\begin{itemize}
  \item \textbf{$> 3000\times$ optimization speedup}
        (${\sim}345$\,s $\to$ $< 0.1$\,s) once the surrogate is
        trained (one-time offline cost of ${\sim}3$--10~hr) and target
        data are generated (${\sim}70$\,s, shared by all methods).
        The surrogate optimization is essentially instantaneous.
  \item \textbf{Complementary accuracy profile}: better on $k_{0,1}$,
        $\alpha_1$, $\alpha_2$; worse on $k_{0,2}$ (7.6\% vs.\ 2.6\%).
  \item \textbf{Implicit regularization} from the RBF interpolation
        smooths noise, making PDE refinement unnecessary and
        potentially counterproductive.
  \item \textbf{Diminishing returns} from additional one-time offline
        training: $3\times$ more samples yielded only 2~pp improvement
        on $k_{0,2}$ with degraded peroxide-current NRMSE.
  \item The \textbf{PDE-based v7 pipeline remains the recommended
        approach} when $k_{0,2}$ accuracy is critical.  The surrogate
        is most valuable for rapid inference (many runs, real-time
        applications) or as a preconditioner for MCMC/EKI sampling.
\end{itemize}

\subsection*{Completed}

\begin{enumerate}
  \item \textbf{Multi-observable joint fitting.}  \textsc{COMPLETED.}
        Fitting total current and peroxide current simultaneously achieved
        the R2 breakthrough: $k_{0,2}$ error 6.1\%, $\alpha_2$ error
        11.7\%.  Multi-observable is the recommended approach for R2
        recovery.

  \item \textbf{Multi-pH inference.}  \textsc{COMPLETED.}  Three-condition
        multi-pH with shared $(k_0, \alpha)$ achieved excellent R1
        ($k_{0,1}$: 6.5\%, $\alpha_1$: 3.6\%) and partial R2
        improvement ($\alpha_2$: 29.5\%).

  \item \textbf{Master inference protocol v1/v2/v3.}  \textsc{COMPLETED.}
        Three iterations of pipeline design converged on a 3-phase
        multi-obs-throughout architecture.  v1's $k_0$-only phase
        poisoned R2; v2's total-current Phase~3 destroyed R2; v3's
        multi-obs-throughout approach recovers all four parameters to
        within 17\%.  Final best: $k_{0,1}$: 16.7\%, $k_{0,2}$: 6.1\%,
        $\alpha_1$: 8.2\%, $\alpha_2$: 11.7\%.

  \item \textbf{Forward solver robustness.}  \textsc{COMPLETED.}
        Mesh refinement ($N_y = 300$, $\beta = 4.0$), smaller timesteps
        ($\Delta t = 0.1$), denser voltage sweeps (19~points), softplus
        regularization, and SNES linesearch tuning raised target
        generation convergence from 78\% to 100\% (57/57).

  \item \textbf{Speed optimization of master pipeline.}  \textsc{COMPLETED.}
        Shared forward solutions for multi-observable phases and predictor
        validation achieved 11.7\% wall-clock speedup (2075\,s $\to$
        1832\,s, ${\sim}30.5$~min) with identical accuracy (all parameter
        errors within 0.03\% of v3).  More aggressive strategies (fewer
        voltage points, skipping Phase~1, relaxed convergence tolerances)
        were tried and reverted due to $> 2\%$ accuracy degradation.

  \item \textbf{Parallelism optimization (v5--v7).}  \textsc{COMPLETED.}
        Three rounds of optimization achieved a 4.4$\times$ cumulative
        speedup (1832\,s $\to$ 415\,s, ${\sim}6.9$~min) while
        \emph{improving} accuracy on all four parameters by
        2.6--5.8 percentage points.  Round~1: parallel fast-path
        dispatch via \texttt{ProcessPoolExecutor} (2.0$\times$).
        Round~2: multi-observable workers computing both adjoint
        gradients per forward solve (3.5$\times$ cumulative).
        Round~3: relaxed optimizer tolerances and pool reuse
        (4.4$\times$ cumulative).  The pipeline is now near the
        practical ceiling for single-machine process-level parallelism;
        further speedup requires MPI Ensemble or GPU acceleration.

  \item \textbf{POD/RBF surrogate model (v8--v9).}  \textsc{COMPLETED.}
        RBF thin-plate spline surrogate trained on 181--445 LHS
        parameter samples, mapping BV parameters directly to I--V curve
        observables.  Reduces optimization time from ${\sim}345$\,s to
        $< 0.1$\,s ($> 3000\times$ speedup), with competitive accuracy
        on $k_{0,1}$ (7.3\%), $\alpha_1$ (4.5\%), and $\alpha_2$
        (4.6\%), but degraded $k_{0,2}$ (9.6\% vs.\ 2.6\%) due to
        peroxide-current interpolation error.  Target generation
        (${\sim}70$\,s) is a shared fixed cost; offline training
        (${\sim}3$--10\,hr) is a one-time amortized cost.
        Key finding: surrogate acts as implicit regularizer, making PDE
        refinement unnecessary.  Diminishing returns observed from
        expanding training data ($3\times$ more samples $\to$ 2~pp
        $k_{0,2}$ improvement at $3\times$ offline cost).  The PDE-based
        v7 pipeline remains the recommended approach when $k_{0,2}$
        accuracy is critical; the surrogate is best suited for rapid
        inference or as a preconditioner for sampling methods.
\end{enumerate}

\subsection*{Future work}

\begin{enumerate}\setcounter{enumi}{7}
  \item \textbf{Combined multi-observable + multi-pH + symmetric.}
        Multi-observable breaks R2 degeneracy (two equations per voltage),
        multi-pH adds concentration diversity ($[\text{H}^+]^2$ scaling),
        and symmetric data constrains $\alpha$ via the anodic Tafel slope.
        Combining all three in a single pipeline could push the R1--R2
        trade-off frontier below the current 11\% maximum error.

  \item \textbf{Neutral-species formulation for wider pH range.}  The
        charged solver fails at $c_{\text{H}^+} > 0.3$ (nondimensional)
        due to Debye-length stiffness.  A neutral-species formulation
        would avoid Poisson stiffness entirely, enabling multi-pH
        inference across the full experimentally relevant range (pH~2
        to pH~12 as in Mangan2025).

  \item \textbf{Gauss--Newton optimizer.}  The current L-BFGS-B
        optimizer uses only gradient information.  A Gauss--Newton
        or Levenberg--Marquardt method could exploit the least-squares
        structure to better condition the $k_0$--$\alpha$ joint
        optimization, potentially breaking the R1--R2 accuracy
        trade-off.

  \item \textbf{Experimental data fitting.}  Apply the validated v7
        pipeline to real RRDE data from Mangan et~al.\ (2025), which
        provides both disk (total) and ring (peroxide) currents---the
        exact observables the pipeline is designed for.  The surrogate
        could accelerate initial parameter sweeps before PDE-based
        refinement.

  \item \textbf{Bayesian uncertainty quantification.}  Replace point
        estimates with posterior distributions via ensemble Kalman
        inversion (EKI) or MCMC to characterize parameter uncertainty
        and the R1--R2 correlation structure.  The surrogate's
        microsecond evaluation time makes MCMC sampling with
        $10^5$--$10^6$ evaluations practical without PDE solves.

  \item \textbf{Transient (chronoamperometric) data.}  R1 and R2 reach
        steady state on different timescales; fitting $I(t)$ at fixed
        overpotentials provides temporal separation invisible in
        steady-state data.

  \item \textbf{MPI Ensemble parallelism for further speedup.}  The
        current \texttt{ProcessPoolExecutor} approach is near its
        ceiling (9~workers on 10~cores, ${\sim}40$\,s pool init
        overhead).  MPI Ensemble parallelism would distribute
        voltage-point solves across MPI ranks, potentially yielding
        $2$--$3\times$ additional speedup on cluster hardware.  This
        requires restructuring from a single-process to an
        \texttt{mpiexec}-launched architecture.
\end{enumerate}
